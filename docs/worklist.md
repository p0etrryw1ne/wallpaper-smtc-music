# WE-SMTC 工作清单

> 本文件记录问题背景、当前处理状态和设计约束。执行前必须先复现或补充测试，再按小步提交实现。

> 2026-06-28 状态：本轮已完成 1、2、4、5、6、7 的代码侧收口并补测试；第 3 条保持为后续架构方向，不在 Wallpaper Engine 页面里假设可直接调用 InfLink-rs 插件对象。

## 1. 封面地址存在但图片加载失败

**现象**

播放器封面区域出现破图图标和空白边框。说明 `thumbnail` 字段非空，前端渲染了 `<img>`，但浏览器实际加载失败。

**当前判断**

- `src/player/renderPlayer.js` 只做字符串层面的 `safeImageUrl()` 过滤。
- `src/media/snapshot.js` 和 `src/we/officialMediaAdapter.js` 只保留原始 `thumbnail` 字符串，没有统一标准化。
- 当前播放器封面没有 `onerror` 回退；坏 `src` 会直接显示破图。
- 旧项目里有 `normalizeThumbnail()` 和 `setImage(... onerror ...)` 两层防护，可作为参考，但不能照搬旧项目结构。

**后续处理方向**

- 增加统一图片源标准化：支持 `data:`、`blob:`、`file:`、`http(s):`、Wallpaper Engine 对象形态、Windows 路径和裸 base64。
- 播放器封面和迷你封面都要有加载失败回退。
- 失败回退应显示现有渐变 fallback，并缓存失败的源，避免每次渲染重复触发破图加载。
- 背景封面也应共享同一套图片源标准化，但不要因为切歌瞬间封面暂缺就立刻切回默认背景。

**当前状态**

- 已新增共享图片源标准化模块，并用于播放器封面与背景。
- 播放器封面、迷你封面已接入图片加载失败回退，破图不再撑出空白边框。
- 已补 `artworkSource`、`artworkDom`、`renderPlayer`、`backgroundState` 相关测试。

## 2. 移除网易云双 SMTC session 合并逻辑

**背景**

最新版 InfLink-rs 插件已解决“网易云官方 SMTC 选项禁用后仍然启动官方 SMTC session”的问题。用户现在只保留 InfLink-rs 的 SMTC 增强，不再需要我们在项目内为网易云双 session 做复杂兜底。

**当前判断**

- 历史上的重复源评分、择优合并逻辑已移除。
- Bridge 现在只过滤空媒体 session，保留 Windows 返回的有意义 SMTC session，不再按 `source_id` 去重。
- 前端源列表保留返回的有意义 session；“切换到下一源”按 distinct `source_id` 循环，因为当前播放控制 API 只能按 `source_id` 定位。

**后续处理方向**

- 继续只保留屏蔽源、低优先级源、手动选择源这些通用策略。
- 不再恢复“同 sourceId 合并/择优”的历史兼容逻辑。
- 用最新版 InfLink-rs + 只启用插件 SMTC 增强的网易云实机验证 session 数量和切源行为。

## 3. 研究 InfLink-rs API，减少 Bridge 承担的网易云专用功能

**依据**

InfLink-rs 文档说明其在网易云/BetterNCM 插件环境中暴露 `window.InfLinkApi`，可读取歌曲、播放状态、时间轴、音量，并提供 `play()`、`pause()`、`next()`、`previous()`、`seekTo()`、`setVolume()` 等控制方法。

**关键限制**

`window.InfLinkApi` 是 InfLink-rs 动态挂载到网易云/BetterNCM 插件运行环境里的对象，不是 Wallpaper Engine Web 壁纸页面的全局对象。壁纸页面不能直接调用它。

**后续处理方向**

- 先确认是否已有可供外部调用的 InfLink-rs 通信通道；没有的话，不应假设 WE 页面能直接访问 `window.InfLinkApi`。
- 如果要利用 InfLink-rs API，应设计一个很薄的网易云侧适配层：
  - 在 BetterNCM 环境中调用 `window.InfLinkApi`。
  - 将当前歌曲、时间轴、播放状态、封面和控制能力暴露给本项目。
  - Bridge 只负责统一对外 HTTP API、源选择和跨播放器兼容，不再承担网易云内部控制细节。
- 对非网易云播放器继续保留通用 SMTC 路径。
- 控制路由建议：
  - 网易云 + InfLink adapter 可用：走 InfLink API。
  - 其他播放器或 adapter 不可用：走现有 SMTC/Bridge 能力。
  - Bridge 未运行：前端继续使用 Wallpaper Engine 官方媒体信息显示，控制能力按官方接口实际能力回退。

## 4. 无 Bridge 服务时封面更新不及时或不更新

**现象**

Bridge 服务未运行时，切歌后封面更新表现不稳定：有时立刻更新，有时延迟更新，有时一直停留在旧封面。

**当前判断**

- 无 Bridge 时前端依赖 Wallpaper Engine 官方媒体接口，封面来自 `wallpaperRegisterMediaThumbnailListener`。
- 当前实现把媒体属性、播放状态、时间轴、封面拆成多个回调合并；切歌时属性回调可能先到，封面回调可能延迟、丢失或返回旧值。
- `wallpaperMediaIntegration.js` 在媒体身份变化时会清掉旧封面，等待后续 thumbnail 回调；如果 WE 没有及时发新封面，播放器就可能出现空封面或旧封面状态不一致。
- 这个问题应与第 1 条区分：第 1 条是“有封面地址但加载失败”，本条是“无 Bridge 时封面事件/数据更新不稳定”。

**后续处理方向**

- 官方媒体回调已增加缩略图竞态保护：新封面早于属性回调到达时不会被后续属性回调误清空。
- 官方媒体路径继续保持与 Bridge 路径分离，Bridge 运行时不覆盖官方回调状态机。
- 后续如仍有实机延迟，应优先补充 WE 回调顺序日志，而不是改背景或播放器渲染层。

## 5. 无 Bridge 服务时进度与歌词滚动存在 0.5-1s 延迟

**现象**

Bridge 服务未运行时，歌词滚动有时比实际播放慢约 0.5-1 秒；启动 Bridge 后歌词同步明显更准确。

**当前判断**

- 无 Bridge 时进度依赖 Wallpaper Engine 官方媒体时间轴回调，时间轴更新频率、延迟和播放器实现有关。
- 启动 Bridge 后，前端使用 Bridge 提供的 SMTC 时间轴快照和本地单调时钟推进，因此歌词更接近真实播放进度。
- 该问题可能不是歌词动画本身变慢，而是无 Bridge 模式下用于选取当前歌词行的 `position` 已经落后。
- 需要与歌词视觉滚动问题区分：视觉滚动是否顺滑是一层，输入进度是否准确是另一层。

**后续处理方向**

- Bridge 已输出时间轴采样时间，前端会映射到页面本地单调时钟，减少 Bridge 路径下的进度漂移。
- 官方媒体 adapter 也保留可选采样时间字段；WE 没有提供时仍按本地时钟推进 playing 状态。
- 后续若实机仍慢半拍，应补真实 WE 回调日志确认源头，而不是加入固定歌词偏移设置。

## 6. A 样式切 B 样式后未落在视觉中心

**现象**

A 样式切换到 B 样式后，播放器整体应落在当前屏幕可用区域的视觉中心，但现在没有贯彻落实。副屏上至少存在视觉偏上的问题。

**当前判断**

- A/B 样式现在只应影响 UI 样式，不应改变媒体源、播放器形态或歌词展开状态。
- 但 B 样式移除了进度条和控制栏后，播放器整体高度变小，如果仍沿用 A 样式的定位或变量，视觉中心会偏移。
- 视觉中心应基于“屏幕顶部到任务栏上沿的可用区域”，而不是只按视口几何中心或按钮点击区域计算。
- 该问题需要和“点击歌曲名临时切换 A/B”逻辑分开处理：逻辑已是 UI-only，剩余问题是布局测量和定位规则。

**后续处理方向**

- A/B 样式已保持为 UI-only：不改变媒体源、不改变播放器形态、不写回 WE 设置。
- B 样式已使用自身视觉高度变量居中，不继承 A 样式包含进度条/控制栏的高度假设。
- 副屏 `2560x1080` 必须列为重点验收；同时覆盖 `3840x2160`、`1920x1080`、`2560x1440`、`3440x1440`。
- 验收时要分别检查：A 展开播放器、A 歌词展开、B 展开播放器、B 歌词展开、迷你播放器。
- 修复时不要让 A/B 样式切换改变 `viewModeOverride`、媒体源选择或歌词展开状态。

## 7. 项目结构与代码质量治理

**现象**

项目经历多轮修补后，部分文件过重、逻辑边界不清晰，存在继续演变成难维护代码的风险。后续修复 1-6 条时，应同步控制结构复杂度，而不是继续在大文件里堆补丁。

**当前重文件**

- `bridge/rust-smtc/src/http.rs`：约 1586 行。HTTP 路由、请求解析、源刷新、选择、控制、歌词接口和测试辅助混在一个文件里。
- `src/app/main.js`：约 748 行。应用状态、Bridge 轮询、WE 回调、命令调度、歌词加载、背景刷新、渲染调度都集中在一个入口。
- `src/styles/base.css`：约 599 行。背景、播放器、迷你播放器、歌词、响应式、A/B 样式混在一份 CSS 中。
- `src/bridge/bridgeClient.js`：约 219 行。Bridge HTTP 请求、超时、命令、源切换和响应归一化都在一起。
- `src/player/renderPlayer.js`：约 191 行。A/B 样式、展开/迷你播放器、控制栏、进度条和封面 HTML 拼接集中在一起。
- `src/lyrics/lyricVisualState.js`：约 166 行。歌词测量、滚动、视觉状态和动画时序紧耦合，后续修改风险较高。

**当前判断**

- 最大问题不是单纯行数，而是职责交叉：媒体源选择、UI 状态、图片加载、歌词时钟、Bridge 控制和布局规则容易互相影响。
- 已废弃需求仍可能残留，例如网易云双 session 合并逻辑；这类历史兼容会干扰当前行为判断。
- 重构不应为了“看起来整洁”大搬家，应优先围绕 1-6 条问题建立清晰边界和测试。

**后续处理方向**

- 前端拆分优先级：
  - 图片源标准化与图片加载失败状态机已独立成小模块，供播放器封面和背景复用。
  - 官方媒体事件合并逻辑与 Bridge 媒体选择逻辑保持分离。
  - 播放器 view mode、A/B style、媒体源选择三类状态继续隔离，禁止互相写入。
  - 歌词“时间源”和“视觉滚动”分层：先得到可信 position，再驱动视觉状态。
  - CSS 按背景、播放器、迷你播放器、歌词、响应式变量拆分，避免所有布局规则堆在 `base.css`。
- Bridge 拆分优先级：
  - `http.rs` 拆出 route handler、source refresh、command dispatch、lyrics endpoint、response serialization。
  - 移除网易云双 session 合并后，删除对应测试和辅助函数。
  - 保留 Bridge 的统一 HTTP API 和跨播放器能力，不把网易云插件内部细节硬塞进通用 SMTC 层。
- 每次结构调整必须是小步、可测试、可回退：
  - 先写或调整测试。
  - 每次只拆一个边界。
  - 每步运行相关测试和 `npm test`。
  - Bridge 改动至少跑 Rust 测试或 release build。
  - 同步 WE 项目目录后再做实机验证。

**禁止事项**

- 不做无测试的大范围搬家。
- 不在 `main.js`、`http.rs`、`base.css` 继续堆新的跨领域逻辑。
- 不保留已由最新版 InfLink-rs 解决的历史兼容代码。
- 不让 UI 样式、播放器形态和媒体源选择继续互相影响。

## 验证要求

- 每个问题必须先补失败测试或可重复手工复现记录。
- 修改后运行 `npm test`。
- Bridge 相关修改必须运行 Rust 测试或至少 release build。
- 同步到 Wallpaper Engine 项目目录后，再做实机验证。
- 不要为已由新版 InfLink-rs 解决的问题继续堆历史兼容代码。
